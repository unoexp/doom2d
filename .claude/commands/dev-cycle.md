# 游戏开发迭代循环

执行一轮完整的游戏开发迭代，包含以下步骤：

## 步骤 1：检查当前状态

扫描 `Assets/_Game/Scripts/` 下所有层的 .cs 文件，整理出每层已实现的模块清单。重点关注：
- 01_Data：数据定义是否完整
- 02_Base：基础设施是否齐全
- 03_Core：核心业务系统是否覆盖
- 04_Gameplay：游戏逻辑层各子系统
- 05_Show：UI表现层
- 06_Extensions：扩展工具
- 07_Shared：共享枚举

## 步骤 2：对比分析

对照以下文档找出缺失模块：
- `docs/程序/Unity 底层程序架构设计方案 v1.0.md` — 架构规划
- `docs/策划/完整游戏设计文档.md` — 游戏设计需求
- 已有代码中的 TODO 注释和空引用

列出所有缺失模块，按优先级排序（核心游戏循环 > 完整体验 > 内容系统）。

## 步骤 3：选择目标

从缺失列表中选择 2-4 个互不依赖的模块作为本轮实现目标。说明选择理由和依赖关系。

**在开始写代码前，向用户展示选择的模块列表并确认。**

## 步骤 4：实现

为每个模块：
1. 确定需要创建/修改的文件和它们所在的层
2. 遵循项目架构约定（五层分层、EventBus 通信、ServiceLocator 注册、中文注释、文件头格式）
3. 编写代码，风格与现有代码一致（区域分隔符、XML文档注释）
4. 如果新系统需要事件，在 `02_Base/EventBus/Events/` 创建对应事件文件
5. 如果需要新枚举，追加到 `07_Shared/Constant/Enums.cs`

## 步骤 5：Review

逐一检查本轮所有新建/修改的文件：
- 编译兼容性：类型引用是否正确，接口是否完整实现
- 架构合规：依赖方向是否正确（低层不引用高层）
- 空引用风险：nullable 操作是否安全
- GC 问题：热路径是否避免了分配（无 LINQ，for 代替 foreach）
- ServiceLocator：Awake 注册 + OnDestroy 注销是否成对
- EventBus：Subscribe/Unsubscribe 是否成对
- 发现问题立即修复

## 步骤 6：提交

将本轮所有改动提交为一个 git commit：
- commit message 使用中文描述本轮实现的模块
- 列出每个模块的关键文件和功能要点
- 末尾加 Co-Authored-By

## 步骤 7：总结

输出本轮迭代结果：
- 已完成的模块列表
- 剩余未实现的模块概览
- 建议下一轮的优先目标
