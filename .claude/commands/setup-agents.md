# Setup Agents — 项目 Agent 生成器

分析当前项目，自动生成适配的 multi-agent 配置。让 AI Company 框架适配任何项目。

---

## 你的任务

扫描当前项目，理解其技术栈和架构，然后生成一套定制化的 agent 定义文件和协调器配置。

## 执行流程

### Step 1: 项目扫描

用 Glob、Grep、Read 工具扫描项目，收集以下信息：

1. **语言和框架**：扫描文件扩展名分布（`.cs`, `.py`, `.ts`, `.go` 等）、包管理文件（`package.json`, `Cargo.toml`, `go.mod`, `*.csproj` 等）
2. **项目结构**：顶层目录布局，识别分层模式（MVC、Clean Architecture、分层架构等）
3. **架构模式**：查找关键文件（路由、中间件、模型、服务、控制器等），识别设计模式
4. **现有配置**：检查 `.claude/` 目录下已有的 agent 和 command 定义
5. **文档**：查找 README、CLAUDE.md、架构文档等

将扫描结果整理为结构化摘要。

### Step 2: 确认项目类型

用 `AskUserQuestion` 向用户确认：

**问题 1**：项目类型确认
- 选项基于扫描结果动态生成（如 "Unity 游戏项目"、"Web 后端项目"、"全栈应用" 等）

**问题 2**：需要哪些角色/部门
- 根据项目类型推荐角色列表，使用 `multiSelect: true`
- 例如游戏项目：架构师、UI实现、数据建模、QA测试、性能优化、美术集成
- 例如 Web 项目：后端工程师、前端工程师、数据库设计、API设计、DevOps、安全审计

**问题 3**：工作流偏好
- 完整流程（会议 + 双重审批）
- 精简流程（单次审批）
- 快速模式（自动执行 + 汇报）

### Step 3: 生成 Agent 定义

根据确认的角色列表，为每个角色生成 `.claude/agents/{role-name}.md` 文件。

每个 agent 定义文件必须包含：

```markdown
---
name: {agent-name}
description: "{触发条件描述，包含具体示例}"
model: {opus|sonnet|haiku}
---

{agent 的系统提示，包含：}
1. 角色定义和专业领域
2. 项目背景（从扫描结果提取）
3. 架构约定（从 CLAUDE.md 或代码规范提取）
4. 工作方法论
5. 输出格式要求
6. 质量标准
```

**model 分配原则**：
- `opus` — 需要深度推理的角色（架构师、总策划）
- `sonnet` — 中等复杂度（实现、QA、会议协商）
- 不使用 `haiku`（Claude Code sub-agent 不支持）

### Step 4: 生成定制化协调器

基于 `.claude/commands/ai-company.md` 的通用模板，生成项目定制版本：

1. 将通用的"部门"替换为项目实际的角色名称
2. 在 Phase 2（专家会议）中指定具体的 agent 组合
3. 在 Phase 6（执行）中添加项目特定的质量检查规则
4. 添加项目特定的架构约定提示

**注意**：如果 `ai-company.md` 已存在，先读取它，然后用 Edit 工具修改，不要覆盖。如果不存在，用 Write 创建。

### Step 5: 生成报告

输出生成报告：

```
## Agent 配置生成报告

### 项目信息
- 类型：{项目类型}
- 技术栈：{语言/框架}
- 架构模式：{识别的模式}

### 生成的 Agent（{数量} 个）
| Agent | 文件 | 模型 | 专业方向 |
|-------|------|------|---------|
| ... | ... | ... | ... |

### 协调器配置
- 文件：.claude/commands/ai-company.md
- 工作流模式：{完整/精简/快速}
- 部门数量：{N}

### 使用方式
运行 `/ai-company {你的目标}` 启动 AI 公司工作流。
```

---

## 重要原则

1. **不假设** — 项目类型和角色必须经过用户确认
2. **不覆盖** — 如果已有 agent 定义，先读取再决定是更新还是新建
3. **适配优先** — agent 的系统提示必须包含项目特定信息，不用通用模板
4. **最小集** — 只生成用户确认需要的角色，不加多余的
